# Puzzle Velocity

Track wall-clock time for lccjs tickets so estimates can be calibrated over time. This Codex skill is repo-local and version-controlled with the lccjs repository.

## When To Use

Use this skill when:

- starting, claiming, or picking up an lccjs GitHub issue
- closing an lccjs issue or preparing a `Closes #N` commit
- setting or checking H/C estimates
- logging a velocity row
- answering project questions about velocity logging

Do not use it for sub-minute clarification turns. Do not log a row for a pure tracker or epic that only collects child issues; children log the work. Scope-decomposition spikes and research deliverables do get their own row.

## Sources Of Truth

Read these repo docs instead of relying on memory:

- `RULES.md` for universal workflow rules
- `docs/claude_workflow.md` for the full issue lifecycle and close protocol
- `puzzle-velocity.md` in `docs` for protocol, role meanings, skip rules, and calibration notes
- `docs/velocity-schema.md` for SQLite columns and valid role values
- `scripts/velocity-log.js` for the insert/export command behavior

The canonical database is `~/.lccjs/lccjs.db`. The committed velocity CSV in `docs` is generated by `npm run velocity:log` / `npm run velocity:export`; never edit it directly.

## Start Protocol

Before reading the issue body or comments, capture the start timestamp:

```bash
date '+%Y-%m-%dT%H:%M:%S%:z'
```

Then confirm the issue is open and claim the worktree according to `docs/claude_workflow.md`.

If the issue has no C estimate, set one before doing substantive work. H is the human discipline budget; C is the agent wall-clock prediction.

## Close Protocol

Before the closing commit, capture the finish timestamp:

```bash
date '+%Y-%m-%dT%H:%M:%S%:z'
```

Run the pre-close error self-audit required by `RULES.md`, log any missing error rows, then log velocity:

```bash
npm run velocity:log -- '{"ticket":N,"role":"DEV","agent":"HONEYDEW","h_min":25,"c_min":15,"actual_min":12,"started_iso":"...","finished_iso":"...","notes":"..."}'
```

`velocity:log` writes to `~/.lccjs/lccjs.db` and exports the committed velocity CSV. The exported CSV must be included in the same closing commit as `Closes #N`.

Leave `closed_commit` empty at close time. Rebases rewrite the commit SHA; derive it later with:

```bash
git log --grep "Closes #N" -1 --format=%h
```

Close via the project script:

```bash
npm run close N
```

## Row Conventions

Use valid `role` values from `docs/velocity-schema.md`: `DEV`, `TEST`, `WRITER`, `RESEARCH`, `SPIKE`, `ARC`, `PM`, `REVIEW`, `COMBO`, `DATA`, or `CHORE`.

Use the terminal/worktree agent name in `agent`, e.g. `HONEYDEW`. Do not invent or backfill unknown model values; leave `model` blank when unknown.

Delta convention:

- `delta_h_min = h_min - actual_min`
- `delta_c_min = c_min - actual_min`

Positive delta means the task finished faster than estimated. Negative delta is valid and means it ran longer than estimated. Do not floor negative deltas to zero.

If `actual_min` is invalid because the session included idle gaps or out-of-scope work, keep the row but leave `actual_min`/delta fields null or annotate the invalidity in `notes`, following `docs/velocity-schema.md`.

## Guardrails

- File a ticket before doing work if none exists.
- Work from a claimed worktree for any repo change.
- Keep one deliverable per close.
- Include the generated velocity CSV in the close commit.
- Do not update calibration analysis inline during ordinary work; file or use a focused DATA/RESEARCH task.
- If a database or logging command fails, log that failure through `npm run error:log` when possible.
