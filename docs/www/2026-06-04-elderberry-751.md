# WWW — ELDERBERRY session 2026-06-04 (#751 Lezer grammar spike / tracker close)

## What went wrong

### 1. `git pull --rebase` without specifying remote/branch
**What happened:** After committing in the worktree, I ran bare `git pull --rebase`. The worktree branch (`elderberry/issue-751-...`) had no upstream tracking information, so git refused with "There is no tracking information for the current branch." Had to retry with `git pull origin main --rebase`.

**Cost:** One failed command, one retry.

**Prevention:** In worktrees created by `npm run claim`, the branch is local-only with no tracking remote. Always use `git pull origin main --rebase` (explicit remote + branch) in worktrees, never bare `git pull`. Add this to the worktree section of `docs/claude_workflow.md` or the claim script's post-claim output.

---

### 2. CSV merge conflict from concurrent agent push
**What happened:** APPLE was working on #792 simultaneously. Between my `git commit` and `git push`, APPLE pushed a velocity row to `docs/puzzle-velocity.csv`. My rebase hit a conflict on that file. I had to inspect the conflict markers, manually resolve (keeping both rows), verify no markers remained, and then `git rebase --continue`.

**Cost:** ~3 extra commands; brief risk of shipping unresolved markers if the verify step had been skipped.

**Prevention:** This is inherent to concurrent multi-agent work. Mitigation: the verify step (`grep -c '^<<<<<<<\|^=======\|^>>>>>>>' <file>` must return 0) is in the protocol and was followed correctly. No structural prevention exists, but the window shrinks if the time between commit and push is short. Don't batch other work between `git commit` and `git push`.

---

### 3. `npm run close` called after manual `git push origin HEAD:main`
**What happened:** I pushed manually with `git push origin HEAD:main`, then ran `npm run close 751`. The close script searches for a closing commit that is NOT yet on the remote. Since I had already pushed, it found nothing and errored: "No unpushed commit references 'Closes #751'." The issue had already been auto-closed by GitHub from the push, so no real harm, but `npm run close` was intended to be the authoritative close path.

**Cost:** The close script failed; had to verify GitHub had auto-closed the issue separately. Minor extra friction.

**Prevention:** This is the same mistake BANANA made on the same day (see `2026-06-04-banana-791.md` error #6). The correct sequence is:
```
git commit -m "... Closes #N"  ← close commit
npm run close N                 ← does rebase + push internally
```
Never `git push` before `npm run close`. If you've already pushed, `gh issue close N` is the fallback. This ordering should be bolded in `docs/claude_workflow.md` — the current docs describe the sequence but don't warn that pushing first breaks the script.

---

### 4. `git worktree remove` failures after rebase
**What happened:** After the rebase and push, I tried to clean up the worktree with `git worktree remove /path/to/elderberry-issue-751` from within the worktree directory. The shell returned `getcwd: cannot access parent directories: No such file or directory` — the process's CWD had become invalid (likely because `npm run close`'s failed run or the rebase moved the HEAD in a way that orphaned the directory). Retrying from the repo root returned `'.claude/worktrees/elderberry-issue-751' is not a working tree` — git no longer registered it.

**Cost:** Two failed commands; unclear whether the directory was ever cleanly removed or just silently orphaned.

**Prevention:**
- Run `git worktree remove` from the **repo root**, not from inside the worktree itself.
- Run it **before** `git push` (i.e. as part of `npm run close`, which handles cleanup). Trying to remove after the push leaves the sequence in an ambiguous state.
- If worktree remove fails with "not a working tree", run `git worktree prune` to let git reconcile its bookkeeping.

---

### 5. Checked #754 alone before batching the rest of the children
**What happened:** When investigating whether #751's children were done, I read #754 first in a solo call, saw it was closed, then batched the remaining three (#752, #753, #755). Minor sequencing inefficiency — all four could have been batched in one call.

**Cost:** One extra round-trip.

**Prevention:** When checking the status of N child issues on a tracker, batch all N in one parallel call. Pattern: `gh issue view A --repo ... && echo "---" && gh issue view B --repo ... && ...`

## Recommendations

### Immediate (docs)
- **`docs/claude_workflow.md`:** Add a callout box in the close sequence section: "Push-before-close is the footgun. `npm run close N` does the push internally. If you have already pushed manually, use `gh issue close N` as fallback." (Both BANANA and ELDERBERRY hit this on 2026-06-04.)
- **`docs/claude_workflow.md`:** In the worktree section, add: "Use `git pull origin main --rebase` (explicit) in worktrees — bare `git pull` fails because worktree branches have no tracking remote."
- **`docs/claude_workflow.md`:** Add: "Run `git worktree remove` from the repo root, not from inside the worktree. If it reports 'not a working tree', run `git worktree prune`."

### Medium-term (skill)
- The `puzzle-velocity` skill's close-protocol section already describes the sequence correctly, but the "do NOT `git commit --amend`" warning overshadows the "push-before-close is broken" warning. Re-order so the push-ordering rule is the first highlighted item.
- Consider whether `npm run close` should detect the "already pushed" case and fall back to `gh issue close N` automatically rather than erroring.
