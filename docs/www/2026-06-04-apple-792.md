# WWW — APPLE session 2026-06-04 (#792 LCC+ trap vector renumber)

## What went wrong

### 1. `npm run claim` run without `--as apple`
**What happened:** First invocation was `npm run claim 792` with no agent identity flag. The script
rejected it immediately: "no agent identity set — pass `--as <fruit>` or export
`CLAUDE_AGENT_NAME=apple`." Had to rerun as `npm run claim 792 -- --as apple`.

**Cost:** One wasted round-trip; minor.

**Prevention:** The `--as <fruit>` flag is always required when the environment variable is not
pre-set. Treat it like a mandatory argument, not an option. Potential script-level fix: if only
one argument is provided and it looks like an issue number, print the required flag as the first
line of the error rather than the generic usage block — the current error is already good, but
the reflex is to reach for `npm run claim <N>` without the flag every time. A memory note or
a `claim` skill snippet showing the full command would help.

---

### 2. `delta_c_min: -5` — wrong sign passed to `velocity:log`
**What happened:** My c_min estimate was 10 min; actual was ~15 min. I computed
`c_min − actual = 10 − 15 = −5` and passed `delta_c_min: -5`. The script rejected it:
"delta_c_min must be >= 0 (got -5); convention is estimate - actual." So the script
*does* want `estimate − actual`, but enforces non-negativity — meaning it only accepts
cases where the estimate was ≥ actual. When actual > estimate the correct value to log
is `0` (under-estimated; delta is non-meaningful rather than negative). I re-logged
with `delta_c_min: 0` and added the explanation to the notes field.

**Cost:** One resubmit; velocity row logged correctly on second attempt.

**Prevention:** The script enforces `>= 0` because a negative delta would mean the estimate
was too low — log `0` and explain in notes. The `velocity-row-field-conventions.md` memory
already flags that the skill's own example row uses the wrong sign; this incident confirms
the trap is real. Add a one-liner to that memory: "when actual > c_min, log delta_c_min=0
and explain the miss in the notes field." Alternatively the script could accept negative
values and just record them as negative deltas — a `0` floor loses information.

---

### 3. `npm run close` before `npm run velocity:log`
**What happened:** After committing the refactor, I ran `npm run close 792` immediately.
The close script's pre-flight check found no velocity row for #792 in the DB and exited:
"Log it via `npm run velocity:log` and amend the commit, then re-run." Had to invoke the
velocity skill, log the row, commit the exported CSV, and then rerun `npm run close`.

**Cost:** One extra commit (the CSV row), one extra close attempt; moderate friction.

**Prevention:** The correct close sequence is:
1. `npm run velocity:log ...`
2. `git add docs/puzzle-velocity.csv && git commit -m "data(velocity): log row for #N ..."`
3. `npm run close <N>`

The sequence is documented in `docs/claude_workflow.md` and the `puzzle-velocity` skill,
but the reflex under task-completion pressure is to reach for `npm run close` as the
first action. A checklist comment at the top of the skill output (or a mandatory
`velocity:log` step baked into `close.js` that prompts rather than exits) would reduce
this. Shorter term: surface the sequence as a single copy-pasteable block in the skill
output every time a puzzle is closed.

## Recommendations

### Immediate (memory)
- Update `velocity-row-field-conventions.md` memory: when actual > c_min, log
  `delta_c_min=0` and explain the undershoot in the notes field — do NOT pass a negative
  value.

### Scripts / tooling
- `close.js`: instead of a hard exit when no velocity row exists, consider printing the
  exact `npm run velocity:log` command pre-filled with the ticket number and role fields
  so the agent can copy-paste rather than context-switch to reconstruct the invocation.
- `claim.js`: if `--as` is missing, make the error message the very first line of output
  (currently it is, but preceded by the `[claim]` prefix — leading with the full correct
  command would be even faster to copy-paste).

### Workflow docs
- `docs/claude_workflow.md` close sequence: add an explicit "DO: velocity:log BEFORE
  npm run close" callout box, mirroring the existing "DO NOT git push manually first"
  callout that was added after the BANANA session.

### Skill
- `puzzle-velocity` skill: at close time, emit the full three-step sequence as a
  copy-pasteable block at the end of its output so it's visible on screen just before
  the agent reaches for `npm run close`.
