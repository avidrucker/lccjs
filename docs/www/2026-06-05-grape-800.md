### 1. Briefing/issue-spec mismatch on #799 not flagged before starting work

**What happened:** The orchestrator briefing for #799 named four patterns to add to `docs/do-this-not-that.md`: CM6 class-name assumption, CDN version-existence assumption, `npm run close` before `velocity:log`, and `delta_c_min` sign/floor handling. **Cost:** No rework was needed (the issue spec was the right thing to implement), but the user was left to discover the mismatch on delivery rather than up front.

**Prevention:** When the orchestrator briefing contradicts the issue body, surface the discrepancy *before* starting work — a single sentence ("your briefing says X, the issue says Y — implementing Y; confirm if wrong") takes seconds and prevents ambiguity from landing silently at close.

### 2. Triage snapshot #813 declared issues "ready" — both closed by concurrent agents within minutes

**What happened:** After running triage, I confirmed #805 and #806 were OPEN and labeled them `severity:medium`. Filed #813 as a priority snapshot. Both issues were closed by other agents within ~10 minutes of the snapshot. When I subsequently declared "#806 is available, want me to pick it up?" and the user agreed, #806 was already closed — discovered only when `npm run claim` was rejected.

**Cost:** One false readiness declaration; one wasted start-timestamp capture; one failed claim attempt.

**Prevention:** In high-throughput orchestration cycles, a triage run goes stale in minutes. Apply `verify-issue-open-before-claiming.md` even when the check was 2 minutes ago: immediately before `npm run claim`, re-run `gh issue view N --json state`. The check→declare→claim window must be zero or near-zero intermediate steps.
