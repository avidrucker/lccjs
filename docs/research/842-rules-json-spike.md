## Q1 — ID scheme

**Chosen: sequential R-prefix (`R001`–`R020`).**

| Scheme | Pro | Con |
|--------|-----|-----|
| `R001`–`R020` | Human-readable in prose ("see R004"), stable, sortable | Sequence gaps if rules are deleted |
| Semantic (`WORKTREE-01`) | Self-documenting category | Requires an upfront stable taxonomy; categories evolve |
| UUID | Globally unique, collision-proof | Unreadable; nobody writes "see b7c4-8f91..." in a doc |

Sequential wins for a small, human-curated rule set. Deletion leaves a gap (R003 is gone) rather than renumbering — that is the desired property. New rules append at the current max + 1.

## Q2 — How does CLAUDE.md load a JSON rules file?

**It can't — directly.** The Claude Code harness reads `.md` files; it has no native JSON import. Three options:

**Option A — JSON is secondary (RULES.md stays authoritative)**  
`RULES.json` is a derived export, regenerated by a script (`npm run rules:export`). Agents and humans read `RULES.md`; tooling reads `RULES.json`. Pro: zero friction to current workflow. Con: two copies diverge unless the export script is run before every commit.

**Option B — JSON is primary (RULES.md is generated)**  
A `scripts/rules-render.js` converts `RULES.json` → `RULES.md` on pre-commit. Agents edit `RULES.json`; `RULES.md` is committed as the rendered output. Pro: single source of truth for IDs/versions/comments. Con: the pre-commit hook must run on every RULES edit; a failed render silently leaves a stale `RULES.md`.

**Option C — JSON companion only, no sync**  
`RULES.json` is a separate artifact that does not try to match `RULES.md` line-for-line. `RULES.md` remains the harness-loaded source; `RULES.json` serves machine consumers (linters, future rule-citation tools). Text divergence is a known, accepted tradeoff. Pro: no tooling required now. Con: text drift must be caught by audit, not enforcement.

**Recommendation: start with Option C.** Build Option B only when a concrete machine consumer (a pre-commit linter, a citation resolver) actually needs it. Premature sync infrastructure is costly to maintain and breaks silently.

## Q3 — When does `version` increment?

**On semantic changes to `text` only, not on `comment` or metadata changes.**

| Change type | Version bumps? |
|-------------|---------------|
| Rule text reworded or tightened | Yes |
| Exception added or removed | Yes |
| `comment` updated or expanded | No |
| `added_issue` backfilled | No |
| `category` reassigned | No |

This makes `version` meaningful: it tracks the rule's *behavioral* history. Version 1 on R016 → R016 v2 in the prototype reflects the actual format refinement after initial introduction. If version were bumped for every metadata touch, it would lose signal.

## Q4 — Can `comment` replace inline rationale in `text`?

**Partially — and the distinction is useful.**

Some rules embed rationale that IS the rule (you cannot strip it without losing the intent):

> R014: "Out-of-scope changes get their own ticket (filed before the close commit), their own worktree, and their own velocity row. Fixes absorbed silently into another ticket's close are invisible to the tracker and corrupt calibration data."

The last sentence is why the rule matters; removing it leaves a bare prohibition with no context for edge cases.

**Heuristic:** `text` should be self-contained enough that an agent can follow the rule correctly without reading `comment`. `comment` holds: historical context, issue citations, recovery procedures, and supplementary examples. When rationale is load-bearing (it affects how you apply the rule), it stays in `text`; when it's explanatory, it moves to `comment`.

Applying this to the 20 rules: 8 rules had rationale that was safely extractable to `comment`; 12 were left nearly unchanged in `text` because the "why" directly shapes correct application.

## Trade-off summary

| Concern | Assessment |
|---------|------------|
| Stable citations (`R004` in prose) | ✅ Big win — renaming/renumbering no longer breaks citations |
| Richer git diffs | ✅ A text change to one rule produces a 1-field diff, not a renumbering cascade |
| `comment` separating rationale from rule | ✅ Useful for audit; mixed for load-bearing rationale (must stay in `text`) |
| CLAUDE.md loading | ❌ Not possible natively; requires a converter or Option C companion model |
| Sync maintenance burden | ❌ Option B (JSON-primary) adds a pre-commit pipeline; Option A adds an export step |
| Human readability during review | ❌ JSON is less readable than Markdown for the primary audience (human + harness) |
| Version tracking | ✅ Per-rule history, not a monolithic file timestamp |

## Recommendation: **conditional go**

Adopt `RULES.json` as an **Option C companion artifact** — machine-readable supplement to `RULES.md`, not a replacement. Immediate value: stable rule IDs for citations and a `comment` field for rationale without bloating `text`. Do not build a sync pipeline until a concrete tooling need (linter, citation resolver, automated rule auditor) justifies the maintenance cost.

If the decision is to move to Option B (JSON-primary), the pre-commit hook in `scripts/git-hooks/pre-commit` must be extended to run `node scripts/rules-render.js` and block on failure. File a separate DEV ticket for that work.
