## 2. Parallel `gh issue create` calls triggered PostToolBatch hook

**What happened:** Filed the `boop` and `ding` issues in a single message with two parallel Bash tool calls. The PostToolBatch hook stopped the response and required serial re-issue.

**Should have:** State-changing Bash calls (anything that writes to GitHub, the filesystem, or git) must always be issued one at a time — even when the calls look independent.

**Prevention:** This is already a standing rule. The hook enforces it. No new doc needed; need to internalise "gh issue create = state-changing, never batch."

## 3. `git rebase --continue --no-edit` — invalid flag

**What happened:** Tried `git rebase --continue --no-edit`. Git rejected it immediately (`--no-edit` is not a valid option for `git rebase`).

**Should have:** Used `GIT_EDITOR=true git rebase --continue` to suppress the editor non-interactively.

**Prevention:** Add to `do-this-not-that.md`: never `git rebase --continue --no-edit`; always `GIT_EDITOR=true git rebase --continue`.

---

## 4. `git add` + `git rebase --continue` chained without `GIT_EDITOR=true`

**What happened:** Ran `grep -c ... && git add docs/puzzle-velocity.csv && git rebase --continue` (for #629 CSV conflict). The `git rebase --continue` at the end tried to open an interactive editor, failed silently, and left the rebase in a confused state. Subsequent re-attempts (including a correct-looking `git add && GIT_EDITOR=true git rebase --continue`) also errored until I staged in a completely separate command.

**Should have:** Done two distinct, sequential commands:
1. `git add <file>` (verify exit 0)
2. `GIT_EDITOR=true git rebase --continue` (separate invocation)

**Prevention:** Add to `do-this-not-that.md`: resolve-and-continue is always two separate Bash calls; chaining them with `&&` risks the rebase seeing a partially-applied state.
