# Errors Log Schema

## Canonical store

**`~/.lccjs/velocity.db`** — SQLite database, local-only, not git-tracked.  
Lives outside the repo; survives `git clean`; accessible from any worktree via absolute path.

The errors table is a sibling of `velocity`. It uses the same conventions (ISO 8601 timestamps, `agent`, `model`, `repo` columns, controlled vocabulary) with one stricter rule: `occurred_iso` is NOT NULL (unlike velocity, where all columns are nullable).

Future CSV export: **`docs/errors.csv`** — will be auto-generated by a future `npm run error:export` command (#899). Do not create this file manually.

---

## Table: `errors`

| Column | Type | Nullable | Description |
|---|---|---|---|
| `id` | INTEGER PK | NO | Surrogate key, auto-incremented by SQLite |
| `occurred_iso` | TEXT | **NO** | ISO 8601 with tz offset captured at the moment of error. Required — an undatable error row is useless for trend analysis |
| `agent` | TEXT | YES | Terminal/worktree name (e.g. `CHERRY`). Nullable: automated hook errors may have no agent identity |
| `model` | TEXT | YES | Claude model short-form (e.g. `sonnet-4.6`). Nullable: non-model errors (hook blocks, bash failures) aren't model-specific |
| `ticket` | INTEGER | YES | GitHub issue active at error time. Nullable: errors outside any puzzle context (e.g. pre-claim failure) have no ticket |
| `repo` | TEXT | YES | Repository context. Default `lccjs`; use `claude-config` for cross-repo ops |
| `error_type` | TEXT | YES | Controlled vocabulary — see valid values below. New rows should always set this |
| `message` | TEXT | YES | Raw error message or first ~200 chars of stderr. Free text; no truncation at insert time |
| `context` | TEXT | YES | JSON blob with type-specific fields (e.g. `{"cmd":"...","exit_code":1}` for BASH_FAIL). JSON keeps schema stable as error types grow; queryable via `json_extract()` |
| `notes` | TEXT | YES | Free-form agent annotation — impact, workaround, resolution |

---

## Valid `error_type` values

| Code | When to use |
|---|---|
| `TOOL_DENIED` | User rejected a tool permission prompt (Bash, Edit, Write, etc.) |
| `HOOK_BLOCK` | pre-commit / commit-msg / pre-push hook exited non-zero |
| `CLAIM_FAIL` | `npm run claim` failed (closed issue, already claimed, missing `--as`) |
| `BASH_FAIL` | Any Bash command exited non-zero with work impact |
| `GIT_FAIL` | `git push`, `git rebase`, `git commit` failed |
| `GH_FAIL` | `gh` CLI / GitHub API error (rate limit, network, not found) |
| `DB_FAIL` | `velocity:log`, `error:log`, or any SQLite operation failed |
| `FILE_FAIL` | Read / Write / Edit tool failure (path not found, permission denied) |
| `SKILL_FAIL` | Skill invocation errored or produced unexpected output |
| `NETWORK_FAIL` | Timeout or connectivity error on web fetch / API call |
| `VALIDATION_FAIL` | Schema validation error (velocity:log field check, etc.) |
| `OTHER` | Uncategorized — use sparingly; prefer a specific type |

## Logging errors

```bash
# Capture the timestamp first
occurred_iso=$(date '+%Y-%m-%dT%H:%M:%S%z')

# Log the row
npm run error:log -- '{
  "occurred_iso": "<occurred_iso>",
  "agent": "CHERRY",
  "model": "sonnet-4.6",
  "ticket": 880,
  "error_type": "BASH_FAIL",
  "message": "git push rejected — remote contains work",
  "context": "{\"cmd\":\"git push origin HEAD:main\",\"exit_code\":1}",
  "notes": "resolved via git pull --rebase"
}'
```

The `error:log` command validates `error_type` against the vocabulary and `model` against the canonical format (`<family>-<major>.<minor>`). Exit 0 on success, 1 on any validation or DB error.

## Context JSON shapes by `error_type`

```jsonc
// BASH_FAIL / GIT_FAIL / GH_FAIL
{"cmd": "git push origin HEAD:main", "exit_code": 1, "stderr": "first 100 chars"}

// TOOL_DENIED
{"tool": "Bash", "cmd_preview": "rm -rf /tmp/foo"}

// HOOK_BLOCK
{"hook": "pre-push", "stderr": "first 100 chars of hook output"}

// CLAIM_FAIL
{"cmd": "npm run claim -- 880 --as CHERRY", "reason": "already claimed"}

// FILE_FAIL
{"tool": "Edit", "path": "/src/core/assembler.js", "error": "ENOENT"}

// DB_FAIL / VALIDATION_FAIL
{"script": "velocity:log", "field": "model", "value": "claude-opus-4-8"}
```

## Querying examples

```bash
# Most recent 10 error rows
sqlite3 ~/.lccjs/velocity.db \
  "SELECT id, occurred_iso, agent, error_type, message FROM errors ORDER BY id DESC LIMIT 10"

# Error counts by type
sqlite3 ~/.lccjs/velocity.db \
  "SELECT error_type, COUNT(*) as n FROM errors GROUP BY error_type ORDER BY n DESC"

# All errors for a specific ticket
sqlite3 ~/.lccjs/velocity.db \
  "SELECT id, error_type, message, notes FROM errors WHERE ticket = 880"

# Errors by a specific agent
sqlite3 ~/.lccjs/velocity.db \
  "SELECT id, occurred_iso, error_type, message FROM errors WHERE agent = 'CHERRY' ORDER BY occurred_iso"

# Errors in a time window
sqlite3 ~/.lccjs/velocity.db \
  "SELECT agent, error_type, message FROM errors
   WHERE occurred_iso >= '2026-06-01' AND occurred_iso < '2026-06-08'"

# Context field extraction (BASH_FAIL commands)
sqlite3 ~/.lccjs/velocity.db \
  "SELECT id, json_extract(context, '$.cmd') as cmd FROM errors WHERE error_type = 'BASH_FAIL'"
```

## Compatibility with velocity table conventions

| Convention | velocity | errors |
|---|---|---|
| Primary key | `id INTEGER PK AUTOINCREMENT` | Same |
| Timestamps | ISO 8601 TEXT with tz offset | Same (`occurred_iso`) |
| Agent column | `agent TEXT` nullable | Same |
| Model column | `model TEXT` nullable | Same |
| Ticket column | `ticket INTEGER` nullable | Same |
| Repo column | `repo TEXT DEFAULT 'lccjs'` | Same |
| Notes column | `notes TEXT` nullable | Same |
| Controlled vocab | `role` TEXT with documented values | `error_type` TEXT with documented values |
| All nullable | Yes | **No — `occurred_iso` NOT NULL** |

## Adding new `error_type` values

1. Add the new code to `VALID_ERROR_TYPES` in `scripts/error-log.js`
2. Add a row to the valid-values table in this file and in `log-error` skill (`~/.claude/skills/log-error/SKILL.md`)
3. No schema migration needed — `error_type` is a plain TEXT column
