# Chapter 16 — Inheritance, Virtual Functions, and Constructors

Source files from the cuh63 package. This chapter covers the three core C++ OOP mechanisms and shows exactly how each is implemented at the machine level.

> **ISA reference:** [docs/lcc-isa.md](../lcc-isa.md)

---

## ex1601.a — Inheritance (static dispatch)

A derived class `B` that inherits from `A`. Both classes have `set` and `display` member functions. Method dispatch is *static* — the compiler knows the type at compile time and calls the correct function directly.

C++ equivalent:
```cpp
class A { public: void set(int n); void display(); protected: int x; };
class B: public A { public: void set(int n, int m); void display(); private: int y; };
```

Layout of a `B` object (2 words, with `x` first — inherited from `A`):
| offset | field |
|--------|-------|
| 0 | `x` (inherited from A) |
| 1 | `y` (B's own field) |

Static dispatch: `aptr->display()` calls `@A@display$v` directly because `aptr` is typed as `A*` at that point — even after `aptr = bptr`. This is the difference from virtual dispatch (ex1602).

Key observation: after `aptr = bptr`, `aptr->display()` prints only `x`, not `x y`, because A's display is called statically.

## ex1602.a — Virtual functions (vtable dispatch)

Same class hierarchy as ex1601, but `A::display` is declared `virtual`. The compiler adds a *vtable pointer* as the first word of each object and dispatches `display` indirectly through it.

Object layout with vtable pointer:
| offset | field |
|--------|-------|
| 0 | vtable pointer |
| 1 | `x` |
| 2 | `y` (B only) |

Vtable setup at allocation time:
```asm
mov r1, 2
bl malloc               ; allocate A object
str r0, fp, -1
lea r1, @A@vtbl
str r1, r0, 0           ; object[0] = &A's vtable
```

Virtual dispatch pattern:
```asm
ldr r0, fp, -1          ; r0 = object pointer
push r0                 ; push this
ldr r0, r0, 0           ; r0 = vtable pointer
ldr r0, r0, 0           ; r0 = vtable[0]  (first virtual function)
blr r0                  ; indirect call
add sp, sp, 1
```

Vtable entries:
```asm
@A@vtbl:  .word @A@display$v
@B@vtbl:  .word @B@display$v
```

Key observation: after `aptr = bptr`, `aptr->display()` now calls `B::display` — because dispatch goes through the *object's* vtable, not the *pointer's* static type.

## ex1603.a — Constructors

Adds constructors to the class hierarchy. Constructors are name-mangled like member functions and called explicitly at object creation. Base constructors are called from derived constructors.

C++ equivalent:
```cpp
class A { public: A(int n); void display(); protected: int x; };
class B: public A { public: B(int n, int m); void display(); private: int y; };
A::A(int n)        { x = n; }
B::B(int n, int m): A(n) { y = m; }
```

Name mangling: `@A@A$i` (A's constructor, one int), `@B@B$ii` (B's constructor, two ints).

**Stack object `A a(1)`:**
```asm
sub sp, sp, 1    ; allocate space for a
mov r0, 1        ; constructor arg
push r0
add r0, fp, -3   ; &a (this)
push r0
bl @A@A$i        ; call A::A(1)
add sp, sp, 2
```

**Base constructor chain in `B::B`:**
```asm
; call A::A(n) from inside B::B:
ldr r0, fp, 3    ; n
push r0
ldr r0, fp, 2    ; this (B object, also serves as A subobject)
push r0
bl @A@A$i
add sp, sp, 2
; then initialize y:
ldr r0, fp, 4    ; m
ldr r1, fp, 2    ; this
str r0, r1, 1    ; this->y = m
```

**Heap object `new A(4)`:**
```asm
mov r1, 1
bl malloc         ; allocate A
str r0, fp, -1    ; store pointer
mov r0, 4         ; constructor arg
push r0
ldr r0, fp, -1    ; push this
push r0
bl @A@A$i
add sp, sp, 2
```
