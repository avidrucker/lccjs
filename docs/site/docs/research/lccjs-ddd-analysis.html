DDD Analysis: avidrucker/lccjs

A thorough critique of the lccjs project through the lens of Domain-Driven Design.

What DDD Actually Requires (Brief Framing)

Domain-Driven Design centers on a few key ideas: identifying the core domain and making it the most protected, expressive part of the codebase; building a ubiquitous language shared by the code and the people who work on it; separating the domain model from infrastructure and application concerns; and using bounded contexts to isolate sub-domains that have their own models and rules.

For LCC.js, the core domain is the LCC ISA — its instructions, machine model, assembly rules, linking contract, and execution semantics. Everything else (file I/O, report formatting, CLI parsing, oracle parity testing) is infrastructure or application layer.

Where LCC.js Does DDD Well

1. The toolchain pipeline maps naturally to bounded contexts, and the project honors that

The three core modules — assembler.js, interpreter.js, linker.js — represent distinct bounded contexts with distinct domain models. Assembly is about transforming source text into machine words through a symbol table. Execution is about simulating a virtual machine with registers, memory, and a fetch-decode-execute cycle. Linking is about resolving cross-module references across object modules. The project keeps these cleanly separated rather than collapsing them into a monolith. That's exactly what bounded contexts call for.

2. The pure API / CLI wrapper split is a meaningful domain/infrastructure boundary

The documented and enforced separation — assembleSource(...) and executeBuffer(...) as pure, stateless seams vs. main(...) as the CLI wrapper that owns file I/O and console output — is the single biggest DDD win in this codebase. DDD calls this separating the domain layer from the infrastructure layer. The project does it consciously, documents it in core.md and each module's docs/ file, and enforces it with typed errors that substitute for CLI exits in the pure path. That's mature practice.

3. Typed errors represent meaningful domain failures

AssemblerError, InterpreterRuntimeError, InvalidExecutableFormatError, LinkerError — these aren't generic JavaScript Error subclasses grabbed for convenience. Each one signals a distinct failure mode in a specific bounded context. That's the DDD principle of making domain concepts explicit in code rather than relying on string-matching error messages or exit codes. The shared location in src/utils/errors.js is a minor architectural awkwardness (see below), but the concept is right.

4. Ubiquitous language is present in the ISA layer

The instruction names (mov, dout, halt, trap, .start, .globl, .word), the file extension conventions (.a, .e, .o, .ap, .ep), and the structural terms (symbol table, load point, object module, executable image) are all used consistently across the codebase, docs, and tests. The docs/lcc-isa.md and docs/lccplus-isa.md serve as the ubiquitous language reference that anchors the whole project. This is exactly what a domain glossary should do.

5. The src/utils separation respects the concept of shared kernel

DDD defines a "shared kernel" as code jointly owned by multiple bounded contexts. reportArtifacts.js, fileArtifacts.js, and errors.js serve all three core modules without belonging to any one of them. The project has correctly identified these as cross-cutting concerns and extracted them, rather than duplicating them inside each core module.

6. Experiments and oracle parity testing have a DDD analog: anti-corruption layers

The experiments/ directory and oracle parity tests (lcc.oracle.e2e.spec.js) are, in DDD terms, an anti-corruption layer — a translation mechanism that mediates between LCC.js's model and the "legacy" oracle LCC binary. Rather than letting oracle behavior silently leak into the LCC.js model, the project explicitly documents deviations in docs/parity_deviations.md. That's disciplined context integration.

Where LCC.js Does DDD Poorly

1. The domain model lives inside large, stateful classes with no explicit model objects

This is the biggest gap. The Assembler class holds the symbol table, the current pass, the line counter, the output buffer, the error list, and the report state all as mutable instance fields. The Interpreter holds memory, registers, PC, and execution state the same way. DDD would push for these to be explicit, named domain objects: a SymbolTable, a MachineState, an InstructionWord, an ObjectModule, an ExecutableImage. Right now those concepts exist only implicitly inside the large classes. You can't pass a SymbolTable to another function — it's always this.symbolTable on the Assembler class. You can't reason about or test a MachineState in isolation — it's tangled into the Interpreter's execution logic.

2. The linker's internal data structures have no domain semantics

From linker.md, the linker manages mca, GTable, ETable, eTable, VTable, ATable. These abbreviated names are the opposite of ubiquitous language — they're implementation artifacts. In DDD terms, the linker's domain model includes concepts like GlobalSymbolTable, ExternalReferenceTable, AddressAdjustmentTable, ObjectModule, RelocatableReference. Using cryptic one-letter-prefixed names obscures the domain and makes the linker a much harder context to work in. The docs acknowledge the linker is "the least refactored core module" — but the problem is conceptual, not just organizational.

3. src/utils/errors.js is in the wrong place for DDD

Typed errors are domain concerns — AssemblerError belongs in the assembler's bounded context, not in a shared utilities folder. Moving them to src/utils implies they're infrastructure, when they're actually the failure vocabulary of the domain. DDD would put each error type in its own bounded context, either inside the module or in an explicit domain/ subfolder. The current arrangement works, but it blurs the line between shared kernel and domain model.

4. The lcc.js orchestrator has no domain concept behind it — it's pure application layer wearing a core module label

lcc.js lives in src/core/ alongside the assembler, interpreter, and linker, but it's fundamentally different — it's an application service, not a domain component. DDD distinguishes between the domain layer (where rules and models live) and the application layer (which orchestrates domain operations for a specific use case). The orchestrator should live in src/app/lcc.js or src/cli/lcc.js to make that distinction visible. Its current placement in src/core/ implies it's coordinate with the assembler and linker, which it isn't — it depends on all of them.

5. The plus variants are a parallel bounded context that's currently a second-class copy

src/plus/assemblerplus.js and interpreterplus.js represent a genuine extension to the domain — the LCC+ ISA adds real new domain concepts (non-blocking input, screen manipulation, timed delays, the .lccplus directive). But the current structure implies these are just "plus-sized" versions of the core modules. DDD would model this as either a separate bounded context (src/plus/ with its own domain model) or as an extension of the core domain via well-defined extension points. Without explicit extension seams in the core, LCC+ is forced to fork the entire assembler/interpreter rather than composing with it. The docs/lccplus-isa.md correctly calls out LCC+ as an addendum, but the code hasn't followed that conceptual separation cleanly.

6. Report generation is partially a domain concern that's been mislabeled as infrastructure

.lst and .bst files aren't just output artifacts — they're part of the domain's observability model, the structured view of an assembly or execution run. genStats.js and reportArtifacts.js sit in src/utils/ but they encode domain knowledge: what fields go in a listing, what column widths mean, how addresses are formatted. A DDD purist would say this is "domain service" logic that happens to produce text, and it should live closer to the assembler/interpreter domain model, not in a shared utilities basket.

7. name.js leaks filesystem state into what should be a pure domain concern

The name.nnn file is essentially the author's name for report headers. name.js resolves it from the current working directory and prompts for it if absent. This is a filesystem-and-interactive-IO concern that's currently tangled with report generation. In DDD terms, the domain concept is just "a UserName string," and the application layer is responsible for resolving it (from a file, a CLI arg, an env var, whatever). The domain shouldn't know or care where the name came from. The existing pure API already offers a workaround (passing userName in options), but name.js pulls the filesystem dependency into a utility that core modules depend on.

Concrete Recommendations for Improvement

1. Extract explicit domain objects from the large classes

Rather than keeping this.symbolTable as a raw object inside Assembler, define a SymbolTable class with methods like define(label, address), lookup(label), hasForwardRefs(). Similarly extract MachineState (registers + memory) from Interpreter, and ObjectModule (header + code bytes + relocation tables) from Linker. These become the core domain model — testable independently, passable between contexts, and named in the ubiquitous language.

2. Rename the linker's internal tables using domain language

Current Name Suggested Name
mca moduleCurrentAddress / allocationPointer
GTable GlobalSymbolTable
ETable / eTable ExternalReferenceTable
VTable VirtualAddressTable
ATable AddressAdjustmentTable

This costs nothing in runtime and buys enormous clarity for anyone working in the linker context.

3. Move lcc.js to src/cli/ or src/app/

This makes the layered architecture explicit:

src/core/     → domain layer (assembler, interpreter, linker)
src/cli/      → application/infrastructure layer (lcc.js orchestrator, CLI wrappers)
src/utils/    → shared kernel (errors, file helpers, report generation)

The orchestrator is an application service, not a peer of the assembler.

4. Consider a src/domain/ layer for pure value objects

An Instruction value object (opcode + operands + encoding), an AssemblyListing (the structured result of a pass, not just a raw buffer), an ExecutionTrace (the sequence of machine states during a run) — these are domain concepts that currently only exist as implicit shapes in return values and internal state. Making them explicit classes with names from the ubiquitous language would bring the codebase much closer to DDD's ideal.

5. Model LCC+ as a composed extension, not a forked parallel

Introduce explicit extension points in the core assembler and interpreter: an InstructionSet interface that the plus variants can extend, a TrapHandler registry that lets new trap implementations register alongside built-ins. Then assemblerplus.js becomes something like:

new Assembler({ instructionSet: lccPlusISA })

...rather than a separate class that reimplements the entire assembly pipeline.

6. Elevate errors.js into the domain layer

Move typed errors to src/core/errors.js (or into each module's own file) and treat them as first-class domain vocabulary. The current src/utils/errors.js placement is a minor but real conceptual misclassification.

7. Make UserName a plain value passed into the application layer

name.js should become an application-layer concern: the CLI resolves name.nnn or prompts before calling any core API, then passes the resolved string in as userName. This keeps the domain model completely free of file I/O and interactive prompts.

Closing Assessment

The project's documentation quality is frankly excellent for a solo/small-team educational project, and it reflects genuine architectural thinking. The main DDD gap is one that's common in toolchain projects: the behavior is well-partitioned but the domain model objects are still implicit, living inside large stateful classes rather than as first-class named types.

Closing that gap — even incrementally, starting with SymbolTable and MachineState — would bring the codebase much closer to what DDD practitioners would recognize as an expressed domain model. The structural bones are good. The next step is making the domain concepts visible as objects, not just as behavior.
