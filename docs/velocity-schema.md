# Velocity Log Schema

## Canonical store

**`~/.lccjs/lccjs.db`** — SQLite database, local-only, not git-tracked.  
Lives outside the repo; survives `git clean`; accessible from any worktree via absolute path.

Exported read-only copy: **`docs/puzzle-velocity.csv`** — committed to git, auto-generated by `scripts/velocity-export.js`. Do not edit the CSV directly.

## Legacy `velocity.db` (deprecated — safe to delete)

The DB was originally named `~/.lccjs/velocity.db`; #947 renamed it to `lccjs.db`
(the name now reflects its multi-table scope: `velocity`, `errors`, `ice_scores`,
and the historical `rice` table — see below). All scripts resolve the path via `scripts/db-path.js`; #984 routed the
last hold-out (`scripts/ice-score.js`) through it too, so **nothing writes
`velocity.db` any more**.

On machines where both files ended up existing, the migration shim left an
orphaned `~/.lccjs/velocity.db` behind. It holds a single `velocity` row
(`ticket=714, GRAPE, RESEARCH`) that is **already present in `lccjs.db`** (id 899),
so it carries no unique data and is safe to remove.

## Table: `velocity`

| Column | Type | Nullable | Description |
|---|---|---|---|
| `id` | INTEGER PK | NO | Surrogate key, auto-incremented by SQLite |
| `ticket` | INTEGER | YES | GitHub issue number (`gh issue view N`) |
| `title` | TEXT | YES | Issue title — short label for readability |
| `role` | TEXT | YES | Work type — see valid values below |
| `h_min` | REAL | YES | Yegor human-effort estimate in minutes (≤60 hard cap; governs decomposition, not forecasting) |
| `c_min` | REAL | YES | Claude wall-clock prediction set *before* starting work (for calibration) |
| `actual_min` | REAL | YES | Elapsed work time in minutes (`finished − started`). **Valid only when the work performed matches the ticket's stated scope** — if out-of-scope work was absorbed (FM-1/FM-2/FM-3 per `docs/research/601-scope-discipline.md`), annotate the row as invalid in `notes` rather than leaving a misleading value (do not delete the row). NULL when the span includes multi-hour idle gaps (multi-turn sessions) — see notes. |
| `delta_h_min` | REAL | YES | `h_min − actual_min` (positive = under human budget) |
| `delta_c_min` | REAL | YES | `c_min − actual_min`. Positive = Claude over-estimated (finished early). Negative = Claude under-estimated (ran over). |
| `started_iso` | TEXT | YES | ISO 8601 timestamp captured *before* `gh issue view` (wall-clock start) |
| `finished_iso` | TEXT | YES | ISO 8601 timestamp captured *before* the closing commit |
| `closed_commit` | TEXT | YES | Always NULL at close time — rebase rewrites the SHA. Derive on demand: `git log --grep "Closes #N" -1 --format=%h` |
| `notes` | TEXT | YES | Free-form. Avoid literal `@todo #N:Est/ROLE` strings (triggers `puzzle:status` scanner). |
| `agent` | TEXT | YES | Terminal/worktree name under which the work ran (e.g. `BANANA`). NULL for pre-#180 historical rows. |
| `model` | TEXT | YES | Claude model short-form used (e.g. `sonnet-4.6`). NULL for rows predating the column (#275). |
| `repo` | TEXT | YES | Repository the `closed_commit` SHA belongs to. Default `lccjs`; set to `claude-config` for cross-repo closes. NULL for rows predating the column (#438). |

## Valid `role` values

| Code | Meaning |
|---|---|
| `DEV` | Implementation / coding |
| `TEST` | Writing or fixing tests |
| `WRITER` | Documentation, comments, protocol docs |
| `RESEARCH` | Investigation, reading, analysis (no code deliverable) |
| `SPIKE` | Scoped research with a concrete finding/decision deliverable |
| `ARC` | Architecture decision |
| `PM` | Project management (filing issues, triage, planning) |
| `REVIEW` | Ratification / sign-off on an artifact someone else produced |
| `COMBO` | Multi-role work in a single session |
| `DATA` | Data analysis, CSV enrichment, notebook re-runs |
| `CHORE` | Housekeeping with no feature/fix value (removing dead code, cleanup) |

## Adding new columns

```sql
ALTER TABLE velocity ADD COLUMN col_name TYPE DEFAULT NULL;
```

Existing rows get NULL automatically. Update `scripts/velocity-export.js` (column order), `docs/velocity-schema.md` (this file), and `docs/puzzle-velocity.md` (protocol).

## Querying examples

```bash
# Most recent 10 rows
sqlite3 ~/.lccjs/lccjs.db \
  "SELECT id, ticket, agent, role, c_min, actual_min FROM velocity ORDER BY id DESC LIMIT 10"

# Rows by role
sqlite3 ~/.lccjs/lccjs.db \
  "SELECT role, COUNT(*) as n, AVG(actual_min) as avg_actual FROM velocity GROUP BY role ORDER BY n DESC"

# Rows for a specific agent
sqlite3 ~/.lccjs/lccjs.db \
  "SELECT id, ticket, role, actual_min FROM velocity WHERE agent = 'BANANA' ORDER BY id"

# C estimate vs actual (calibration)
sqlite3 ~/.lccjs/lccjs.db \
  "SELECT ticket, role, c_min, actual_min, ROUND(c_min * 1.0 / actual_min, 2) as ratio
   FROM velocity WHERE actual_min IS NOT NULL AND actual_min > 0 ORDER BY id DESC LIMIT 20"
```

## Notes on `actual_min`

For multi-turn sessions with human review time between turns, the raw
`finished_iso − started_iso` delta includes idle gaps (potentially hours) and
is not a meaningful measure of active work time. In these cases, leave
`actual_min`, `delta_h_min`, and `delta_c_min` NULL and note the reason in
the `notes` field. Keep `h_min` and `c_min` (estimates are still valid).

`actual_min` is also invalid as a calibration data point when the elapsed time
includes work outside the ticket's stated scope — e.g. an unplanned bug fix absorbed
mid-session (FM-1: bug tax), a triage pass that ran longer than ~5 minutes (FM-2:
discovery bleed), or a scope creep addition (FM-3). In these cases the measured
duration reflects a mix of deliverables, making it appear the C estimate was
wrong when it may have been accurate for the in-scope work alone. When this occurs:
**do not delete the row** — annotate `notes` with the out-of-scope description (e.g.
`"actual_min invalid — absorbed FM-1 fix for #N, ~10 min untracked"`) so the
corruption is visible in the data rather than silent. See
`docs/research/601-scope-discipline.md` for the full FM taxonomy.

## Table: `ice_scores` (live — current prioritization)

ICE scores for open issues. Populated via `npm run ice:score`. Superseded the `rice` table in #956.

**Formula:** `ICE = I × C / E` where I = Impact (1–3), C = Confidence (0.0–1.0), E = Effort (hours).

```sql
CREATE TABLE IF NOT EXISTS ice_scores (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  issue          INTEGER NOT NULL UNIQUE,
  title          TEXT,
  type           TEXT,
  I              REAL,       -- Impact (1 = minor · 2 = significant · 3 = flagship/critical)
  C              REAL,       -- Confidence (0.0–1.0)
  E              REAL,       -- Effort in hours (H_min / 60)
  ice_score      REAL,       -- I * C / E
  tier           TEXT DEFAULT '',   -- 'critical' | 'elevated' | ''
  yegor_priority INTEGER,
  actionable     TEXT DEFAULT 'Y',  -- 'Y' | 'N'
  labels         TEXT,
  notes          TEXT,
  updated_iso    TEXT
);
```

### Related files

- `scripts/ice-score.js` — interactive + batch scoring, tier assignment, export to CSV + Markdown
- `stats/ice-scores.md` — current scores table (auto-generated by `npm run ice:score`)
- `npm run ice:score` — the live command; see script for full flag reference

## Table: `rice` — HISTORICAL (superseded by ICE, #956)

> **Historical.** RICE was the issue-prioritization model before #956 introduced
> ICE (`npm run ice:score`, `scripts/ice-score.js`, `stats/ice-scores.md`, DB table
> `ice_scores`). **ICE is the live prioritization table** — use `ice:score`, not the retired
> `rice:*` scripts. The `rice` table and `stats/rice-scores.csv` are kept as a
> historical data source (`ice-score.js` can one-time-seed from the RICE CSV). The
> `rice-seed.js` / `rice-log.js` / `rice-export.js` scripts were retired to
> `scripts/archive/` and their npm scripts removed (#997); the schema below is
> documented for reading existing rows only.

RICE scores for open issues. Was populated via the retired `rice:seed` (seeded from `stats/rice-scores.csv`) / `rice:log` scripts.

```sql
CREATE TABLE rice (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  issue          INTEGER NOT NULL,
  title          TEXT,
  type           TEXT,
  r              REAL,      -- Reach (1–4 scale)
  i              REAL,      -- Impact (0.25/0.5/1/2/3 scale)
  c_pct          REAL,      -- Confidence % (30–100)
  e_hrs          REAL,      -- Effort in hours (H_min / 60)
  rice_score     REAL,      -- r * i * (c_pct/100) / e_hrs
  rice_rank      INTEGER,
  yegor_priority INTEGER,
  actionable     TEXT,      -- 'Y' | 'N'
  labels         TEXT,
  notes          TEXT,
  scored_iso     TEXT,
  scored_by      TEXT,
  repo           TEXT DEFAULT 'lccjs'
);
```
