# CLAUDE.md — `stats/` (data-analysis workspace)

Guidance specific to doing **data analysis / data science** in this project (Jupyter
notebooks, `enrich.py`, the velocity/RICE/ICE/errors datasets). The root
[`../CLAUDE.md`](../CLAUDE.md) owns the toolchain and the multi-agent workflow — this
file only adds what's different when you're analyzing data here. Read both.

## What lives here

`stats/` analyzes the project's own operational data — primarily the Yegor-style
puzzle-velocity log, plus RICE/ICE scoring and the agent error log. See
[`README.md`](./README.md) for the file-by-file inventory and how to run the notebooks.

## Environment: use the shared venv, never a per-project one

Data-science tooling (jupyter / pandas / numpy / scipy / matplotlib / seaborn) is
installed **once, as a single source of truth**, via the dotfiles `datascience` section:

```bash
~/dotfiles/install.sh datascience          # one-time (or to update tooling)
```

This provisions the shared venv `~/.venvs/datasci` and puts `jupyter` on PATH. Run
notebooks with `~/.venvs/datasci/bin/python` or just `jupyter lab`.

- **Do NOT create a per-project venv** for ad-hoc analysis here — use the shared workbench.
- The SSOT package list is `~/dotfiles/python/datascience-requirements.txt`. Edit it and
  re-run the section to add a package; don't `pip install` into the venv by hand.
  (`requirements.txt` here documents the deps but the shared venv is the runtime.)

## DB safety — destructive ops need explicit human go-ahead

The DB is shared by concurrent agents. **Never `DELETE` or run a destructive `UPDATE`
on `~/.lccjs/lccjs.db` (or any DB) without explicit permission from the user.**

1. Always `SELECT` first to show exactly which rows are affected.
2. Wait for an explicit go-ahead before mutating.

"Run it yourself" for a prior command does not authorize a later destructive DB op. This
is RULES.md rule 7 (`../RULES.md`). Auto-mode blocks these by default; getting it
unblocked still requires the SELECT-first + consent steps above.

## Notebook commit convention

Commit notebooks **with their embedded outputs and plots** so they render on GitHub
(the day-N analysis notebooks are committed this way on purpose). Don't strip outputs.
Re-running a notebook before committing keeps the rendered version honest.

## Refreshing a weekly notebook

The `week-NN-analysis.ipynb` files are **window-filtered and committed** — "copy last
week's notebook and re-run it" is *not* the same as "analyze all the data." Three footguns
bit the week-04 pass (#1519) and a naive re-run will silently repeat them:

1. **The date `CEILING` is hardcoded — edit it.** The load cell pins the window with a literal
   like `CEILING = '2026-06-14'` (a settled week-end from the notebook you copied). Re-executing
   keeps the *old* cutoff, so new weeks silently never appear. Grep the load cell for the ceiling
   and bump it (e.g. `CEILING → '2026-06-29'`) before you trust anything downstream.
2. **`nbconvert --execute` can exit 0 while a cell raised — scan cells, don't trust the exit
   code.** `jupyter nbconvert --execute --inplace` returned **0 while a cell threw a `TypeError`**;
   the traceback was in stdout but the exit code was clean and the inplace file was left stale.
   Verify a run by scanning the executed notebook for error outputs
   (`output_type == 'error'`), not by checking `$?`.
3. **Inherited analysis code breaks at scale.** A §8 line — `sorted(err_df['hst_day'].unique())` —
   worked at **138** error rows and threw `TypeError` (mixed NaN/str) at **390**. Code you copied
   carries latent type assumptions that only fail once the dataset grows; expect and re-test the
   inherited cells against the larger window.

Source: #1519 / the 2026-06-29 BANANA TIL in [`../docs/learnings/`](../docs/learnings/).

## Logging velocity rows from analysis work

If your analysis work is itself a logged puzzle, `npm run velocity:log` has gotchas the
generic skill example gets wrong:

- **`role`** must come from the fixed set: DEV, TEST, WRITER, RESEARCH, SPIKE, ARC, PM,
  COMBO, DATA, CHORE. Data-analysis work is usually **DATA**. A bad value only *warns*,
  then silently lands — get it right up front.
- **`model`** must be canonical short-form — `opus-4.8`, `sonnet-4.6` — **not** the full
  ID `claude-opus-4-8`. `tests/new/puzzle-velocity-csv.unit.spec.js` enforces `/^[a-z]+-\d+\.\d+$/`.
- **`--from-main`** is required (and must come *after* the JSON arg) when logging outside a
  worktree; from inside a worktree, omit it.
- **`delta_c_min`** may be negative — log the real value, don't floor to 0.

Full field reference: [`../docs/velocity-schema.md`](../docs/velocity-schema.md).
